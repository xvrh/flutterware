# TUI Stage 4.5a — Focus system (tree, manager, traversal)

**Status:** Draft
**Date:** 2026-05-16
**Scope:** Add the focus tree, focus manager, `Focus`/`FocusScope` widgets,
focus traversal, and key-event routing to the TUI framework — the first half
of the "interactivity" stage. The declarative `Actions`/`Intents`/`Shortcuts`
key-binding layer is stage 4.5b, a separate spec.

**Extends:** [`2026-05-15-tui-stage4-widget-layer-design.md`](./2026-05-15-tui-stage4-widget-layer-design.md)

---

## Context

Stage 4 delivered the reactive widget layer: `Widget`/`Element`/`State`,
`setState`, `InheritedWidget`, `RenderObjectWidget`, `BuildOwner`, `runApp`.
The framework can build real screens declaratively, but it has no notion of
**focus**: there is no "which widget is active," and key events are not
routed. The widget layer only *exposes* the raw `terminal.keys` stream through
the `TerminalApp` `InheritedWidget`; a `StatefulWidget` that wants keyboard
input subscribes to that stream itself (see `widget_showcase.dart`'s
`_keySub` / `_subscribed` wiring in `didChangeDependencies`).

That pattern does not scale to an interactive TUI app — lists, tabs, focus
traversal — because every focusable widget would receive every keystroke and
there is no way to express "this panel is focused; send keys here." Stage 4
deliberately deferred a focus system as "its own later stage."

The roadmap's standing decision is "reimplement the engine; transcribe the
framework." Focus is framework-shaped: the focus-node tree, the manager, and
traversal are rendering-backend agnostic and full of hard-won edge cases, so
they are transcribed closely from `package:flutter`'s `focus_manager.dart`,
`focus_scope.dart`, and `focus_traversal.dart`. The one place the
transcription adapts is the key model — see "Divergences from Flutter" below.

Stage 4.5a delivers a usable, testable, demo-able focus system on its own:
after it, a screen can have focusable panels, Tab/arrow traversal, and a
visible focus highlight. Stage 4.5b then layers `Actions`/`Intents`/
`Shortcuts` on top, and Stage 5 builds the real flutterware CLI app.

## Goals

1. **`FocusNode` / `FocusScopeNode`** — a focus tree running parallel to the
   element tree: parent/children links, `hasFocus`/`hasPrimaryFocus`,
   `canRequestFocus`/`skipTraversal`, `enclosingScope`, `requestFocus()`,
   `unfocus()`. `FocusScopeNode` groups children and remembers its
   most-recently-focused descendant.
2. **`FocusManager`** — owned by `TuiBinding`. Holds `rootScope` and
   `primaryFocus`, applies pending focus changes at the start of each frame,
   and dispatches key events down the focus chain.
3. **`Focus` / `FocusScope` widgets** — `StatefulWidget`s that own (or adopt) a
   node, attach it to the parent node found via a `_FocusMarker`
   `InheritedWidget`, and re-expose `hasFocus` to descendants. `Focus.of` /
   `FocusScope.of` ancestor lookups.
4. **Focus traversal** — `FocusTraversalPolicy` with `ReadingOrderTraversalPolicy`
   (reading order by post-layout geometry) and `DirectionalFocusTraversalPolicy`
   (arrow keys), and a `FocusTraversalGroup` widget that scopes a policy to a
   subtree.
5. **Key-event routing** — `runApp` pumps `terminal.keys` into
   `FocusManager.handleKeyEvent`, which walks the focus chain
   (primary → ancestors → root) calling each node's `onKeyEvent`, stopping on
   `KeyEventResult.handled`. Tab/Shift-Tab and arrow keys that fall through
   hit a **built-in traversal fallback**.
6. **A demo** (`focus_demo.dart`) — a 2×2 grid of focusable panels with
   Tab/arrow traversal and a visible focus highlight, plus a `FocusScope`.

## Non-goals (for this round — 4.5b or later)

- **No `Actions` / `Intent` / `Action` / `Shortcuts` / `ShortcutManager`.** The
  declarative key-binding layer is stage 4.5b. In 4.5a the Tab/arrow traversal
  fallback lives directly in `FocusManager`; 4.5b replaces that block with
  `Shortcuts`→`Actions` routing. This is the deliberate seam between the two.
- **No animation.** No `Ticker`, `SchedulerBinding`, or `AnimationController`.
  Deferred beyond the interactivity stage entirely.
- **No `FocusableActionDetector`** (it depends on `Actions`) and no mouse /
  pointer focus — a TUI has no pointer.
- **No `LogicalKeyboardKey` / `HardwareKeyboard` / key-down-up-repeat model.**
  The TUI keeps its existing single-`KeyEvent`-per-press model — see
  "Divergences from Flutter."
- **No `GlobalKey`.** Carried forward from stage 4; not needed here.
- **No repaint boundaries / layer model.** Unchanged from stage 4.
- **No changes to the stage 1–3 engine, paint kit, or render objects** other
  than one additive, read-only global-offset helper on `RenderObject` (and, if
  `terminal.keys` is not already a broadcast stream, making it one).

## Architecture

```
Widget tree            Element tree                  Focus tree
(immutable)            (mounted; BuildContext)        (FocusNode / FocusScopeNode)
───────────            ───────────────────────        ───────────────────────────
RootWidget        →    RootElement                    rootScope ◀─ FocusManager
  …               →      …                                │        (TuiBinding-owned)
  FocusScope      →      StatefulElement ────────────▶ FocusScopeNode
    Focus         →        StatefulElement ──────────▶   FocusNode  ★ primary
      <child>     →          …RenderObjectElement
    Focus         →        StatefulElement ──────────▶   FocusNode
      <child>     →          …RenderObjectElement

terminal.keys ─▶ FocusManager.handleKeyEvent(KeyEvent)
                   → focus chain: primary ▶ parent ▶ … ▶ rootScope
                   → call node.onKeyEvent along the chain; stop on `handled`
                   → unhandled Tab/arrows → built-in traversal fallback
```

The focus tree is a second tree, parallel to the element tree, linked to it
only through the `Focus`/`FocusScope` widgets that own the nodes. A `Focus`
widget finds its parent focus node through a `_FocusMarker` `InheritedWidget`
inserted by the nearest enclosing `Focus`/`FocusScope`, and re-exposes its own
node the same way. `hasFocus` therefore propagates to descendants over the
existing stage 4 inherited-widget dependency path — no new reactive primitive.

New code lives under `app/lib/src/tui/widgets/` as new `part` files of the
existing `widgets.dart` library, so the focus classes share library-private
element lifecycle state — the same organization as `render/` and the rest of
`widgets/`.

## `FocusNode` / `FocusScopeNode` — `focus_manager.dart`

```dart
enum KeyEventResult { handled, ignored, skipRemainingHandlers }

typedef FocusOnKeyEventCallback = KeyEventResult Function(
    FocusNode node, KeyEvent event);

class FocusNode {
  FocusNode({this.skipTraversal = false, bool canRequestFocus = true});

  FocusNode? _parent;
  final List<FocusNode> _children = [];
  FocusManager? _manager;

  bool skipTraversal;
  bool get canRequestFocus;          // false also disables descendants' default
  set canRequestFocus(bool value);

  FocusOnKeyEventCallback? onKeyEvent;

  bool get hasFocus;                 // this node or a descendant is primary
  bool get hasPrimaryFocus;          // this node IS primary
  FocusScopeNode? get enclosingScope;
  FocusScopeNode get nearestScope;
  Iterable<FocusNode> get ancestors;
  Iterable<FocusNode> get descendants;

  void requestFocus([FocusNode? node]);   // make this (or node) primary
  void unfocus();                          // hand focus to the enclosing scope

  // tiny listenable — NOT package:foundation ChangeNotifier
  void addListener(void Function() listener);
  void removeListener(void Function() listener);
  void _notify();

  void dispose();
}

class FocusScopeNode extends FocusNode {
  FocusScopeNode();
  FocusNode? _focusedChild;          // most-recently-focused descendant
  FocusNode? get focusedChild;
  bool get isFirstFocus;
  void setFirstFocus(FocusScopeNode scope);
}
```

`FocusNode` is **a minimal listenable**: a private `List<void Function()>` plus
`addListener`/`removeListener`/`_notify`. It does not extend
`package:flutter/foundation`'s `ChangeNotifier` — the roadmap's zero-pub-
dependency rule stands. The `Focus` widget's `State` adds a listener and calls
`setState` when the node notifies.

Attaching a node to the tree (`_parent` / `_children` mutation) and detaching
it is done by the `Focus`/`FocusScope` element lifecycle, not by callers.

## `FocusManager` — `focus_manager.dart`

```dart
class FocusManager {
  FocusManager();
  final FocusScopeNode rootScope = FocusScopeNode();
  FocusNode? get primaryFocus;

  // set by TuiBinding: schedule a frame when focus changes while idle
  void Function()? onFocusChange;

  void _markNeedsUpdate();           // a focus change is pending
  bool get _hasPendingChange;
  void applyFocusChangesIfNeeded();  // called at the top of drawFrame

  KeyEventResult handleKeyEvent(KeyEvent event);
}
```

- **Pending focus changes.** `FocusNode.requestFocus`/`unfocus` record the
  desired next primary focus and call `FocusManager._markNeedsUpdate`, which
  invokes `onFocusChange` (the binding schedules a frame). The change is
  *applied* by `applyFocusChangesIfNeeded`, called at the top of
  `TuiBinding.drawFrame` **before** `buildOwner.buildScope`. Applying it
  updates `primaryFocus`, updates each affected scope's `_focusedChild`, and
  `_notify()`s every node whose `hasFocus` changed. Those notifications run
  `Focus` states' `setState`, landing them in the same frame's dirty list.
- **Key dispatch.** `handleKeyEvent` builds the focus chain — `primaryFocus`
  then its ancestors up to `rootScope` — and calls `node.onKeyEvent(node,
  event)` along it. `handled` and `skipRemainingHandlers` stop the walk;
  `ignored` continues to the next ancestor. If the whole chain returns
  `ignored`, the **built-in traversal fallback** runs: `SpecialKeyCode.tab`
  with no shift → `policy.next`; with shift → `policy.previous`; arrow keys →
  `policy.inDirection`. The fallback returns `handled` if it moved focus.

## `Focus` / `FocusScope` widgets — `focus_scope.dart`

```dart
class Focus extends StatefulWidget {
  const Focus({
    super.key,
    required this.child,
    this.focusNode,                  // adopt an external node, else own one
    this.autofocus = false,
    this.onKeyEvent,
    this.canRequestFocus = true,
    this.skipTraversal = false,
  });
  final Widget child;
  final FocusNode? focusNode;
  final bool autofocus;
  final FocusOnKeyEventCallback? onKeyEvent;
  final bool canRequestFocus;
  final bool skipTraversal;

  static FocusNode of(BuildContext context);
  static FocusNode? maybeOf(BuildContext context);

  @override
  State<Focus> createState() => _FocusState();
}

class FocusScope extends Focus {
  const FocusScope({ ... this.node ... });
  // owns/adopts a FocusScopeNode
  static FocusScopeNode of(BuildContext context);
}
```

`_FocusState`:

- `initState` — adopt `widget.focusNode` or create a `FocusNode`; copy
  `onKeyEvent`/`canRequestFocus`/`skipTraversal` onto it; add a listener that
  `setState`s.
- `didChangeDependencies` — read the parent node from the ancestor
  `_FocusMarker`; (re)attach this node under it; if `autofocus` and the
  enclosing scope has no focused child yet, `requestFocus()`.
- `didUpdateWidget` — push changed fields onto the node; re-parent if the
  ancestor node changed.
- `dispose` — detach the node; dispose it only if this widget created it.
- `build` — wraps `child` in a `_FocusMarker` (an `InheritedWidget` carrying
  this node), so a descendant `Focus` finds this as its parent and a
  descendant that reads `Focus.of(context)` rebuilds when `hasFocus` flips.

`_FocusMarker` is the private `InheritedWidget`; `updateShouldNotify` is true
when the node identity or its `hasFocus` changes. `Focus.of` does the standard
`dependOnInheritedWidgetOfExactType<_FocusMarker>()` lookup.

## Focus traversal — `focus_traversal.dart`

```dart
enum TraversalDirection { up, down, left, right }

abstract class FocusTraversalPolicy {
  FocusNode? findFirstFocus(FocusNode currentNode);
  bool next(FocusNode currentNode);
  bool previous(FocusNode currentNode);
  bool inDirection(FocusNode currentNode, TraversalDirection direction);
  Iterable<FocusNode> sortDescendants(Iterable<FocusNode> descendants);
}

class ReadingOrderTraversalPolicy extends FocusTraversalPolicy { ... }
class DirectionalFocusTraversalPolicy extends FocusTraversalPolicy { ... }

class FocusTraversalGroup extends StatelessWidget {
  const FocusTraversalGroup({super.key, required this.policy,
      required this.child});
  final FocusTraversalPolicy policy;
  final Widget child;
  static FocusTraversalPolicy of(BuildContext context);
}
```

Traversal works on **post-layout geometry**: a focusable node's screen rect is
its `Focus` element's nearest descendant render object's global offset and
size. Traversal runs between frames, so the previous frame's layout is current.

- `ReadingOrderTraversalPolicy` — the default. Collects the focusable,
  non-`skipTraversal` descendants of the current `nearestScope`, sorts them by
  rect in reading order (top row first, then left column), and `next`/
  `previous` step through that order, wrapping at the ends.
- `DirectionalFocusTraversalPolicy` — `inDirection` picks the nearest focusable
  whose rect lies in the requested direction from the current node's rect
  (band-overlap preferred, then Euclidean distance), used for arrow keys.

`FocusTraversalGroup` exposes a policy to its subtree via an `InheritedWidget`;
`FocusManager`'s fallback reads the policy nearest the current primary focus,
defaulting to `ReadingOrderTraversalPolicy` when no group is present.

The "global offset of a render object" needed for rects is a small read-only
helper added to `RenderObject` — it walks parent offsets to the root. It does
not change layout or paint behavior.

## `TuiBinding` / `runApp` integration — `binding.dart`

- `TuiBinding` gains `final FocusManager focusManager = FocusManager();`.
  `focusManager.onFocusChange` is wired to the binding's frame scheduler (the
  same `onFrameNeeded` path `BuildOwner.onBuildScheduled` already uses).
- `TuiBinding.drawFrame` calls `focusManager.applyFocusChangesIfNeeded()` at
  the top, **before** `buildOwner.buildScope`, so focus-driven `setState`s land
  in the frame being built.
- `runApp` subscribes to `terminal.keys` once and forwards each event to
  `binding.focusManager.handleKeyEvent`. `terminal.keys` must be a broadcast
  stream (the `TerminalApp` `InheritedWidget` still exposes it and the existing
  showcase still listens); if it is not already broadcast, that is a one-line
  engine fix.
- `TerminalApp` is unchanged — the raw `keys` stream stays available for
  opt-out consumers. Focus routing is purely additive.

## The demo — `app/examples/tui/focus_demo.dart`

A 2×2 grid of `Focus` panels. Each panel is a bordered box whose border is
bright/double-line when `Focus.of(context).hasFocus` is true and dim otherwise,
and shows its `hasFocus` state as text. Tab/Shift-Tab walk the reading order;
arrow keys do directional traversal; `q` quits via `TerminalApp.of(context).
exit()`. One panel pair is wrapped in a `FocusScope` to exercise scoped
grouping. One panel uses `autofocus: true` so the demo starts with a focused
panel.

## Divergences from Flutter

Recorded so the transcription is intentional, not accidental:

- **Single key event, no key lifecycle.** A terminal delivers one "key
  pressed" event; there is no key-down/key-up/repeat and no `HardwareKeyboard`.
  `onKeyEvent` takes the existing `KeyEvent` (which already carries
  `Set<Modifier>` and value equality). Flutter's `KeyEventResult` enum is
  transcribed unchanged.
- **No `LogicalKeyboardKey`.** Key identity is the existing `CharKey`/
  `SpecialKey` model. This matters in 4.5b (`Shortcuts` activators key off
  `KeyEvent` directly, no `LogicalKeySet`); 4.5a only needs `SpecialKeyCode.tab`
  and the arrow codes for the fallback.
- **`FocusNode` is a hand-rolled listenable**, not `ChangeNotifier` — zero pub
  dependencies.
- **Focus changes apply in `drawFrame`**, not via `SchedulerBinding` persistent
  frame callbacks (there is no scheduler binding until the deferred animation
  stage).
- **Traversal geometry comes from the render tree directly** via a global-
  offset helper, rather than Flutter's `RenderObject.getTransformTo` /
  `semantics`-driven rects.

## Testing strategy

The whole focus system is testable without a tty, the same way stages 3 and 4
are: a `TuiBinding` driven against a `CellBuffer` wrapped in a `Painter`, with
`KeyEvent`s fed directly into `FocusManager.handleKeyEvent`. Build a widget
tree with `Focus` panels, call `drawFrame`, assert on `primaryFocus`, on
`hasFocus` reaching dependents (cells read back from the buffer), and on
traversal moving focus as expected.

Key cases:

- **`FocusNode`** — attach/detach maintains parent/children; `enclosingScope`/
  `nearestScope` resolve correctly; listeners fire exactly on `hasFocus`
  change; `dispose` detaches and stops notifications.
- **`FocusManager`** — `requestFocus` sets the pending change and
  `applyFocusChangesIfNeeded` promotes it to `primaryFocus`; `unfocus` hands
  focus to the enclosing scope; the focus chain is primary→ancestors→root;
  `handleKeyEvent` calls `onKeyEvent` down the chain and stops on `handled`.
- **Traversal** — reading order sorts by rect (row then column);
  `next`/`previous` wrap at the ends; `skipTraversal` nodes are excluded;
  `inDirection` picks the geometrically nearest node; `FocusTraversalGroup`
  scopes its policy.
- **`Focus` widget** — owns a node when none is passed and adopts one when it
  is; `autofocus` focuses on mount when the scope is empty; a descendant
  reading `Focus.of` rebuilds when `hasFocus` flips and not otherwise;
  `didUpdateWidget` pushes changed fields.
- **End-to-end routing** — a simulated `KeyEvent` reaches the focused node's
  `onKeyEvent`; an unhandled Tab moves focus via the fallback; the next
  `drawFrame` paints the new focus highlight.

The only thing requiring a real terminal is `focus_demo.dart`, covered by
manual smoke.

## Success criteria

1. All existing tests still pass; `flutter analyze` is clean.
2. Every new `app/test/tui/widgets/focus_*_test.dart` suite passes.
3. `dart tool/prepare_submit.dart` produces no diff.
4. `focus_demo.dart` runs in a real terminal:
   - The 2×2 grid renders; one panel starts focused (`autofocus`).
   - Tab/Shift-Tab move focus in reading order, wrapping at the ends; the
     focused panel's border highlight follows.
   - Arrow keys move focus directionally.
   - The `FocusScope`-wrapped pair groups as expected.
   - `q` calls `TerminalApp.of(context).exit()`; `runApp` returns and the
     terminal is restored cleanly.
